```java
public class Menu  {
    // 2. 필드(속성) 정의 : 상품 번호, 상품 이름, 상품 설명
    private String index;
    private String name;
    private String description;

    // 4. 메소드 정의: 각 상품에 접근 가능한 getter 메소드
    public String getIndex() {
        return index;
    }
    public String getName() {
        return name;
    }
    public String getDescription() {
        return description;
    }
    // 중략
```

## Getter Setter를 사용하는 이유

Getter, Setter와 같은 `엑세스 함수` 사용 시, `데이터 무결성`과 `유효성 검사`가 가능해진다.

예를 들어, 외부에서 index이라는 필드에 직접 접근한다면 0보다 낮은 값을 줄 수도 있기 때문에 이 경우 객체의 무결성이 깨지게 된다.

- 그렇다면 무결성이란 무엇일까?
  - 객체의 무결성이란 데이터의 정확성과 일관성을 유지하고 보증하는 것을 말한다.
- 그렇다면 Getter/Setter를 어떻게 데이터 무결성이 지켜질까?
  - Getter, Setter를 이용해서 데이터를 생성 및 접근을 하게 되면 들어오는 값을 바로 저장하는게 아닌, 한번 검증하고 처리할 수 있도록 할 수 있기 때문에 데이터의 무결성이 지켜진다.

객체의 필드를 `private`과 같은 `접근제한자`를 두면서 객체지향의 목적인 `정보은닉`이 가능해진다.

- Getter : 본 필드의 값을 숨긴 채 내부에서 가공된 값을 꺼낼 수 있도록 하는 것
- Setter : 필드를 private로 만들어 외부의 접근을 제한한 후, Setter를 사용해 전달받은 값을 내부에서 가공해 필드에 넣어줄 수 있도록 하는 것

### **추가로_1**

무분별한 setter는 무결성을 해칠 수 있다.
setter로 데이터를 수정하면, 어떤 부분을 수정하는지, 어디서 데이터를 수정하는지 알 수 없다.

### **추가로_2,**

이런 맹점을 위해 `Builder()` 를 사용하는데

Builder 패턴이란 복잡한 객체를 생성하는 방법을 정의하는 클래스와 표현법을 정의하는 클래스를 분리하여, 서로 다른 표현이라도 이를 생성할 수 있는 동일한 절차를 제공하는 패턴이다.

setter대신 각각의 필드를 생성하는 메소드를 작성해서 안전하게 사용하는 방법으로 보다 안전하게 사용할 수 있다.

각 매개변수를 넣어주는 메서드의 리턴 자료형을 Builder 객체로 지정함으로써, 메서드 체이닝 기법을 적용하고, 매개변수를 다 넣은 경우 build() 메서드로 객체를 생성합니다.

이렇게 빌더 패턴을 사용하면 어느 필드에 어떤 값을 채워야 할지 명확하게 알 수 있으며, 데이터 일관성, 객체 불변성 등을 만족시키며 코드의 가독성 또한 올라갑니다.
하지만 빌드 패턴은 구현시 많은 코드양이 필요하고, Builder라는 객체를 하나 더 만드는 것이기 때문에 사용하는 코드에 따라 성능이 낮아질 수 있습니다.

따라서 클래스를 설계할 때 필수, 선택 인자들이 많은 경우 Builder 패턴을 사용하는 것이 효율적입니다.

### **추가로_3,**

- 스프링에서는 `Lombok` 라이브러리를 사용해 빌더 패턴을 사용할 클래스에 @Builder 어노테이션 하나만 적용해줌으로써 빌더 패턴을 사용할 수 있게 됩니다.
